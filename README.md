# ZenPulse: AI Meditation App (Prototype)

Прототип мобильного приложения для медитаций с системой подписки (Paywall), списком сессий и AI-генерацией аффирмаций.

---

## 🚀 Стек

- Expo + React Native
- Expo Router (file-based routing)
- react-native-safe-area-context
- Архитектура: FSD (pages / features / entities)
- AI: мок-LLM (асинхронная генерация с задержкой)

---

## 📱 Реализованный функционал (по ТЗ)

### 1️⃣ Экран Paywall
- Премиальный темный дизайн
- Список преимуществ Premium
- Два тарифа (Месячный / Годовой)
- Годовой визуально выделен как выгодный
- Кнопка «Попробовать бесплатно» имитирует успешную покупку

---

### 2️⃣ Экран Медитаций (Главный)

- Список карточек (название + длительность)
- Free и Premium контент
- Locked-логика:
  - если `isSubscribed = false` и медитация Premium → карточка серая + 🔒
  - при нажатии на locked → переход на Paywall
- После “покупки” Premium карточки становятся активными

Условие блокировки:
```ts
const locked = item.isPremium && !isSubscribed;
```

---

### 3️⃣ AI «Настрой дня»

- Выбор настроения (3 смайлика)
- Кнопка генерации
- Асинхронная генерация текста (имитация LLM запроса)
- Loading-состояние ("Генерирую...")

## 🧭 Routing

Используется Expo Router:

- `/` → экран медитаций
- `/paywall` → экран подписки

Глобальные провайдеры подключены в `app/_layout.tsx`.

---

## 🔐 Логика подписки

Подписка реализована через React Context:

- `isSubscribed` — флаг доступа
- `subscribe()` — имитация покупки
- `reset()` — вспомогательная dev-функция

Locked-логика реализована на уровне карточек, а не глобальным редиректом, что соответствует требованиям ТЗ.

---

## 🏗 Архитектура (FSD)


Бизнес-логика и UI разделены по FSD.

## 🤖 Как ИИ справился с мобильной спецификой

ИИ позволил быстро собрать базовую структуру экранов, но мобильная адаптация потребовала итерационной доработки.

### Навигация
- Использован Expo Router (file-based routing)
- Изначально была сгенерирована вкладочная структура (Tabs)
- Для соответствия ТЗ навигация была упрощена до Stack
- Переход на Paywall реализован через `router.push("/paywall")`

### SafeArea и layout
- В корневом layout подключён `SafeAreaProvider`
- Экраны обёрнуты в `SafeAreaView`
- Paywall дополнительно обёрнут в `ScrollView`, чтобы CTA была доступна на маленьких экранах

### Ручная корректировка
После генерации UI были вручную скорректированы:
- padding и gap
- размеры карточек
- визуальное состояние locked-контента
- поведение layout на маленьких устройствах

ИИ ускорил создание структуры и UI, однако финальная мобильная адаптация потребовала ручной проверки и корректировки.

## ❓ Контрольный вопрос

**С какими специфическими проблемами мобильной верстки ИИ справляется хуже всего и как я контролировал его работу, чтобы приложение не сломалось на маленьких экранах (iPhone SE vs Pro Max)?**

ИИ хуже всего справляется с мобильными особенностями, которые зависят от реального размера устройства и поведения layout:

1. **Вертикальное переполнение (overflow по высоте)**  
   На маленьких экранах (например, iPhone SE) контент может не помещаться, а ИИ часто генерирует layout без учета scroll-контейнера.

2. **SafeArea (чёлка, статусбар, нижние отступы)**  
   Без явного использования SafeAreaView элементы могут залезать под системные области.

3. **Непропорциональные отступы и размеры**  
   ИИ склонен задавать фиксированные padding и размеры, которые выглядят хорошо на большом экране (Pro Max), но перегруженно — на маленьком.

4. **Навигационные группы (Stack vs Tabs)**  
   Изначально была сгенерирована вкладочная структура, которая не соответствовала требованиям ТЗ, и её пришлось упростить до Stack-навигации.

---

### Как я контролировал это:

- Использовал `SafeAreaProvider` и `SafeAreaView` в корневом layout
- Обернул Paywall в `ScrollView`, чтобы кнопка CTA всегда была доступна
- Проверял экран на разных размерах (маленький vs большой)
- Убирал фиксированные высоты и корректировал padding/gap вручную
- Тестировал кликабельность нижних элементов
- Упростил навигацию, убрав Tabs и лишний layout-слой

ИИ ускорил создание базовой структуры, но финальная мобильная адаптация потребовала ручной проверки и корректировки.
